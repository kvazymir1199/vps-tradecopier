This package is a **Next.js app** (not a published library), so the sync uses a few
repo-specific arrangements. Read this before re-syncing.

## Build arrangement (package shape, no dist)

- There is **no library `dist/`**. We bundle from a generated barrel entry instead:
  - `cfg.entry = .design-sync/.cache/entry.tsx` â€” generated by `.design-sync/build-entry.mjs`,
    which `export *`s every module under `src/components` (so all components AND their
    compound sub-parts land on `window.TradeCopierUI`) plus re-exports `toast` from sonner.
  - `cfg.componentSrcMap` lists the **top-level** components that get preview cards
    (18 today). Sub-parts (DialogContent, SelectItem, TableHeader, â€¦) stay in the bundle
    but are not carded â€” that is intentional.
- **Tailwind v4 CSS** is compiled to a static stylesheet by `.design-sync/build-css.mjs`
  (PostCSS + `@tailwindcss/postcss`) into `.design-sync/.cache/compiled.css`, which
  `cfg.cssEntry` points at. The raw `src/app/globals.css` only declares tokens + `@import
  "tailwindcss"`; without compilation the components render unstyled. `build-css` scans
  `src/**` and `.design-sync/previews/**` (via `@source` in `.design-sync/tw-input.css`) so
  utility classes used by previews are emitted too.
- `cfg.buildCmd = node .design-sync/build-css.mjs && node .design-sync/build-entry.mjs` â€”
  always run this (or the driver, which runs it) before `package-build.mjs`.
- `.d.ts` contracts come from `cfg.dtsPropsFor` (hand-written, types inlined since the
  emitted `.d.ts` only imports React). Synth/barrel mode can't extract inline prop types
  from `.tsx`, so these are authoritative â€” keep them in sync if a component's props change.

## arm64 native binaries (IMPORTANT on a fresh clone / re-install)

This machine is **win32-arm64**, but `npm install` here pulled the **x64** native binaries
for `lightningcss` and `@tailwindcss/oxide`. Under arm64 node, `build-css.mjs` then fails
with `Cannot find module '../lightningcss.win32-arm64-msvc.node'`. Fix (does NOT touch the
lockfile): install the matching-version arm64 platform packages into `node_modules`:

```
npm pack lightningcss-win32-arm64-msvc@<lightningcss ver> @tailwindcss/oxide-win32-arm64-msvc@<oxide ver>
# extract each tarball's package/ dir to node_modules/<name>
```

(Versions must match the installed `lightningcss` / `@tailwindcss/oxide`.) If the repo is
ever reinstalled under x64 node, this won't be needed.

## Render check / playwright

- `playwright` (JS only, no bundled chromium) is installed under `.ds-sync` with
  `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1`.
- The render check uses the **system Chrome** via `DS_CHROMIUM_PATH`:
  `DS_CHROMIUM_PATH="C:/Program Files/Google/Chrome/Application/chrome.exe" node .ds-sync/package-validate.mjs ./ds-bundle`
  (also pass it to `package-capture.mjs`). No 200 MB chromium download needed as long as
  Chrome stays installed at that path.

## Preview conventions

- Previews import from `"frontend"` (the package name) â†’ redirected to `window.TradeCopierUI`.
  Icons import from `lucide-react` (bundled normally). `toast` imports from `"frontend"`
  (re-exported so it shares the bundled Toaster's sonner instance).
- **Overlay components** (Dialog, AlertDialog, Select, and all the *Dialog composites,
  MappingsPanel) use `cfg.overrides.<Name> = {cardMode:"single", viewport:"WxH"}` so the open
  state renders inside the card. Changing any `cfg.overrides` requires a **full
  `package-build.mjs`** (scoped `preview-rebuild` trips `[CONFIG_STALE]`).
- **Data-bound composites** (LinksTable, TerminalsTable, MappingsPanel) fetch
  `http://localhost:8000/api/*`. Their previews patch `window.fetch` at module top to return
  canned data so the cards show realistic rows offline. The fetch-mock data shapes mirror
  `src/types/index.ts` and the hook endpoints â€” if those endpoints/shapes change, update the
  mock in the matching `.design-sync/previews/<Name>.tsx`.

- `[RENDER_THIN] Toaster` â€” toasts live in a fixed/portal layer, so the measured root height

- `tokens: 2 missing (below threshold)` â€” non-blocking; a couple of `var(--*)` referenced by

## Re-sync risks (what can silently go stale)

- **The source was changing during the first sync.** Two components (`AddTerminalDialog`,
  `EditMagicMappingDialog`) plus edits to `terminals-table`/`mappings-panel` appeared on disk
  mid-run (timestamps 2026-06-21 14:01). If the frontend keeps evolving, re-run the driver to
  pick up added components â€” but **new top-level components must be added to
  `cfg.componentSrcMap` and `cfg.dtsPropsFor` by hand** (discovery is componentSrcMap-driven
  in barrel mode; a new `src/components/*.tsx` lands in the bundle via the barrel but is NOT
  carded until listed).
- **`cfg.dtsPropsFor` is hand-maintained** â€” it does not auto-track prop changes in source.
  Re-check a component's contract if its props change.
- **Fetch-mock data is inlined in the table previews** and mirrors the API shapes â€” it rots if
  the backend contract changes.
- **arm64 binary fix is gitignored** (`node_modules`) â€” re-apply on a fresh clone/reinstall.
- **Tailwind compile depends on `@tailwindcss/postcss` + native oxide/lightningcss** â€” a
  toolchain bump may need re-matched arm64 binaries.
